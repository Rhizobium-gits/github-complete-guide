# 01. Gitとは何か

## バージョン管理とは

プログラムや文書の変更履歴を記録・管理する仕組みのこと。

**バージョン管理がない場合:**
```
レポート_最終版.docx
レポート_最終版_修正.docx
レポート_最終版_修正2_これが本当の最終版.docx
レポート_提出用_本当にこれで最後.docx
```

**バージョン管理がある場合:**
```
レポート.docx  ← 1つのファイルに全ての変更履歴が記録されている
  ├── v1: 初稿作成 (2024-01-10)
  ├── v2: 第2章を追加 (2024-01-15)
  ├── v3: 図表を修正 (2024-01-20)
  └── v4: 最終校正 (2024-01-25)
```

## Git と GitHub の違い

| | Git | GitHub |
|---|-----|--------|
| **何か** | バージョン管理ソフトウェア | Gitリポジトリのホスティングサービス |
| **動作場所** | ローカル（自分のPC） | クラウド（インターネット上） |
| **開発者** | Linus Torvalds（Linuxの作者） | GitHub社（現Microsoft傘下） |
| **料金** | 完全無料 | 無料プランあり（有料プランもあり） |
| **役割** | ファイルの変更履歴を記録 | Gitの履歴をオンラインで共有・協業 |

```
あなたのPC (Git)                    インターネット上 (GitHub)
┌─────────────────┐    push →     ┌──────────────────────┐
│  ローカル        │              │  リモートリポジトリ    │
│  リポジトリ      │    ← pull    │  (みんなが見れる)      │
│  (.git フォルダ) │              │  github.com/user/repo │
└─────────────────┘              └──────────────────────┘
```

## 基本的な考え方

### リポジトリ (Repository / repo)

プロジェクトのファイル群と、その変更履歴の全てを保存する場所。フォルダのようなもの。

- **ローカルリポジトリ**: 自分のPCにあるリポジトリ
- **リモートリポジトリ**: GitHub上にあるリポジトリ

### コミット (Commit)

ファイルの変更を「保存」する操作。ゲームのセーブポイントのようなもの。

```
コミット1 → コミット2 → コミット3 → コミット4 (現在)
"初期作成"   "機能A追加"  "バグ修正"   "機能B追加"
```

各コミットには以下が記録される：
- **誰が** 変更したか（Author）
- **いつ** 変更したか（Date）
- **何を** 変更したか（Diff）
- **なぜ** 変更したか（Commit Message）
- **一意のID**（ハッシュ値: `a1b2c3d...`）

### ブランチ (Branch)

開発の「枝分かれ」。メインの開発ラインに影響を与えずに、別の作業ができる。

```
main:        A → B → C ──────────→ F (マージ)
                   \              /
feature:            D → E ───────
                    "新機能の開発"
```

### ステージング (Staging)

コミットする前に、「どのファイルの変更を含めるか」を選ぶ作業。

```
作業ディレクトリ          ステージングエリア          リポジトリ
(Working Directory)      (Staging Area / Index)    (Repository)
┌──────────────┐         ┌──────────────┐          ┌──────────────┐
│ ファイルを編集 │  add →  │ コミット対象を │ commit → │ 履歴に記録    │
│              │         │ 選んで準備    │          │              │
└──────────────┘         └──────────────┘          └──────────────┘
```

## Gitの歴史

### Gitの誕生 (2005年)

| 年 | 出来事 |
|----|--------|
| **2005年4月** | Linus Torvalds（Linuxカーネルの作者）がGitを開発開始 |
| **2005年4月** | わずか**10日間**で最初のバージョンが完成 |
| **2005年6月** | GitでLinuxカーネルの管理を開始 |
| **2005年12月** | Git 1.0 リリース |

**なぜGitが作られたのか：**

LinuxカーネルはそれまでBitKeeperという商用ツールで管理されていた。2005年にBitKeeperの無償利用が終了したため、Linus Torvaldsは自分で新しいバージョン管理システムを作ることを決意。以下の要件で設計された：

- **高速**: 大規模プロジェクトでもサクサク動く
- **分散型**: サーバーなしでもローカルで完全に動作する
- **データの完全性**: 一度記録したデータが壊れない
- **非線形な開発**: 何千ものブランチを並行管理できる

> 「Git」という名前はLinusの自己嘲笑（英語スラングで「嫌なやつ」の意味）とも、"Global Information Tracker" の頭文字とも言われている。

### バージョン管理の歴史

```
1972年  SCCS（最初のバージョン管理システム）
  ↓
1982年  RCS（ファイル単位の管理）
  ↓
1990年  CVS（ディレクトリ単位、中央集権型）
  ↓
2000年  Subversion / SVN（CVSの後継）
  ↓
2005年  Git（分散型の革命）← ここが転換点
  ↓
現在    Git が事実上の標準（シェア95%以上）
```

**中央集権型 vs 分散型：**

```
中央集権型 (SVN等):              分散型 (Git):
     ┌──────┐                 ┌──────┐   ┌──────┐
     │サーバー│                 │ PC A │   │ PC B │
     │(唯一) │                 │(完全)│   │(完全)│
     └──┬───┘                 └──┬───┘   └──┬───┘
   ┌───┼───┐                    │          │
   ↓   ↓   ↓                    ↓          ↓
  PC  PC  PC              ┌──────┐   ┌──────┐
 (部分)(部分)(部分)           │GitHub│   │ PC C │
                          │(完全)│   │(完全)│
サーバーが死んだら全滅        └──────┘   └──────┘
                          誰のPCにも完全な履歴がある
```

### GitHubが変えた世界

**1. オープンソースの民主化**

GitHub以前、OSSへの貢献は難しかった（メーリングリストでパッチを送る等）。GitHubのFork & Pull Requestの仕組みが、誰でもワンクリックで貢献できる世界を作った。

**2. ソーシャルコーディング**

GitHubはプログラマーの「SNS」としても機能。スター、フォロー、コントリビューショングラフ（草）が開発者同士のつながりを生んだ。

**3. 世界を動かすプロジェクトのホスト**

GitHubでホストされている有名プロジェクト：

| プロジェクト | 説明 | スター数 |
|------------|------|---------|
| **Linux** | OSカーネル | 180k+ |
| **React** | Facebook製UIライブラリ | 225k+ |
| **VS Code** | Microsoft製エディタ | 165k+ |
| **TensorFlow** | Google製機械学習 | 185k+ |
| **Python (CPython)** | Python言語本体 | 65k+ |
| **Kubernetes** | コンテナオーケストレーション | 110k+ |
| **Bitcoin** | 暗号通貨 | 80k+ |

**4. 「履歴書」としてのGitHub**

技術者の採用面接で「GitHubアカウントを見せてください」が当たり前に。コントリビューショングラフ（草）、公開リポジトリ、OSSへの貢献が技術力の証明になった。

### GitHubの競合サービス

| サービス | 特徴 |
|---------|------|
| **GitLab** | セルフホスト可能、CI/CD統合、DevOps全般 |
| **Bitbucket** | Atlassian製（Jira/Confluenceと連携）、少人数チーム向け |
| **Gitea** | 軽量セルフホスト型、Goで書かれている |
| **Azure DevOps** | Microsoft製、エンタープライズ向け |
| **SourceHut** | ミニマリスト志向、メーリングリストベース |

> GitHubが圧倒的シェア（推定90%以上）を持つが、組織のポリシーでGitLabやBitbucketを使うケースもある。

## Git/GitHubを使うメリット

1. **変更履歴が全て残る** — いつでも過去の状態に戻れる
2. **複数人で同時作業** — ブランチで独立して開発し、後で統合
3. **バックアップ** — GitHubにコードが保存されるので安心
4. **コードレビュー** — プルリクエストで他の人にチェックしてもらえる
5. **オープンソース** — 世界中の人とコードを共有・協力できる
6. **ポートフォリオ** — GitHubプロフィールが技術力の証明になる
