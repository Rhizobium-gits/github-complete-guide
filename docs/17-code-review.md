# 第17章　コードレビュー — 品質を高め合う文化

## 17.1 この章で学ぶこと

ソフトウェア開発において、コードを書くことと同じくらい重要なプロセスがあります。それが**コードレビュー**です。コードレビューとは、あるメンバーが書いたコードを別のメンバーが読み、フィードバックを返す活動のことです。

コードレビューは単なるバグチェックの手段ではありません。チーム全体の技術力を向上させ、知識を共有し、コードベースの一貫性を保つための文化的なプラクティスです。世界中の優れたソフトウェアチームが、コードレビューを開発プロセスの中核に位置づけています。

この章を読み終えると、以下のことができるようになります。

- コードレビューの目的と価値を深く理解する
- GitHubのPull Request上でレビューを実施する具体的な手順を身に付ける
- ブランチ保護ルールを設定して、レビューを開発プロセスに組み込む方法を知る
- レビュアーとしての効果的なフィードバックの書き方を習得する
- PR作成者として、レビューしやすいPull Requestを作成するコツを学ぶ

---

## 17.2 コードレビューとは

### なぜレビューするのか

コードレビューの目的は多岐にわたります。ひとつずつ見ていきましょう。

**バグの早期発見**は、コードレビューの最も分かりやすい効果です。書いた本人は自分のロジックに没頭しているため、見落としやすいミスがあります。別の視点を持つ人が読むことで、nullチェックの漏れ、境界値の処理ミス、セキュリティの脆弱性などを本番環境に入る前に発見できます。本番環境でバグが見つかった場合の修正コストは、開発段階で見つかった場合の数倍から数十倍になると言われています。

**知識の共有**も重要な目的です。チーム開発では、特定の機能を特定のメンバーだけが理解している状態（「バス係数が1」と呼ばれます）はリスクです。コードレビューを通じて、チームの全員がコードベースの異なる部分を理解する機会が生まれます。レビュアーは他のメンバーが書いたコードを読むことで新しい技法を学び、レビューを受ける側も別の視点からのフィードバックによって成長できます。

**コード品質の向上**は、レビューの継続的な効果です。自分のコードが他の人に読まれると分かっていると、自然とより読みやすく、より丁寧なコードを書くようになります。また、レビューを通じてチームの「良いコードとは何か」という基準が共有され、時間とともにコードベース全体の品質が向上していきます。

**一貫性の維持**も見逃せません。大きなプロジェクトでは、複数のメンバーがそれぞれ異なるスタイルでコードを書くと、コードベースの可読性が低下します。レビューを通じて命名規則、エラーハンドリングのパターン、ディレクトリ構成などの一貫性を保つことができます。

### コードレビューは「検査」ではなく「対話」

コードレビューを効果的に行うためには、マインドセットが大切です。レビューは上司が部下のコードを「検査」するものではなく、チームメンバー同士が「より良いコードにするために対話する」プロセスです。経験の浅いメンバーが書いたコードでも、ベテランが書いたコードでも、レビューを受けることには等しく価値があります。

## 17.3 レビューの進め方

GitHubでは、Pull Requestの画面上でコードレビューの一連の作業を行うことができます。ここでは、レビュアーとしての具体的な操作手順を解説します。

### Files changedタブ

Pull Requestのページには複数のタブがありますが、レビューの中心となるのは「**Files changed**」タブです。このタブを開くと、Pull Requestに含まれるすべてのファイル変更が差分（diff）形式で表示されます。

<!-- screenshot: Files changedタブの差分表示 -->

差分表示では、**緑色の背景**が追加された行、**赤色の背景**が削除された行を示しています。行が変更された場合は、変更前の行が赤色で、変更後の行が緑色で表示されます。

大きなPull Requestの場合、すべてのファイルを一度に見るのは大変です。左側のファイルツリーを使って、特定のファイルにジャンプできます。また、各ファイルの横にある「Viewed」チェックボックスを使って、レビュー済みのファイルをマークしておくと、後でどこまで確認したかが分かりやすくなります。

### コメントの追加方法

差分表示で気になる箇所を見つけたら、その行にコメントを追加できます。コメントを追加したい行の行番号にカーソルを合わせると、青い「+」アイコンが表示されます。これをクリックするとコメント入力欄が開きます。

複数行にまたがるコメントを書きたい場合は、開始行の「+」アイコンをクリックしたまま終了行までドラッグします。これにより、コードの範囲を指定してコメントできます。

### コメントの種類

GitHubのレビューコメントには、いくつかの種類があります。

**通常のコメント**は、質問、指摘、提案などを自由に書けます。

```markdown
ここのロジックですが、userがnullの場合にNullPointerExceptionが
発生する可能性がありそうです。nullチェックを追加した方が安全ではないでしょうか。
```

**suggestion（提案）**は、GitHubの強力な機能のひとつです。具体的なコード修正案をコメントに含めることができ、PR作成者はワンクリックでその修正を適用できます。コメント入力欄のツールバーにある提案ボタンをクリックするか、以下のように手動で記述します。

````markdown
nullチェックを追加することで安全になります。

```suggestion
if (user != null) {
    return user.name;
}
return "Unknown";
```
````

PR作成者の画面では、このsuggestionに対して「Apply suggestion」ボタンが表示され、クリックするだけでコードが修正されます。小さな修正であれば、PR作成者がわざわざローカルに戻って修正する手間が省けるため、非常に効率的です。

**質問コメント**は、コードの意図や設計判断について確認するためのものです。

```markdown
この変数名の `ctx` は何の略でしょうか？
ドメイン固有の用語であれば、もう少し明示的な名前にした方が
新しいメンバーにも分かりやすいかもしれません。
```

### レビュー結果の送信

個々のコメントを書き終えたら、レビュー全体の結論を送信します。画面右上の「**Review changes**」ボタン（または「**Finish your review**」ボタン）をクリックすると、3つの選択肢が表示されます。

**Comment（コメント）**は、承認でも拒否でもない中立的なフィードバックです。「いくつか質問がありますが、全体的には良さそうです」のような場合に使います。

**Approve（承認）**は、変更を承認する意思表示です。「このPull Requestはマージして問題ない」という判断を示します。ブランチ保護ルールで必須レビュー数が設定されている場合、必要な数のApproveが集まるまでマージできません。

**Request changes（変更を要求）**は、マージ前に修正が必要であるという意思表示です。セキュリティ上の問題や、明らかなバグ、設計上の大きな懸念がある場合に使います。Request changesが付いたPull Requestは、そのレビュアーがApproveに変更するか、レビューをdismissするまでマージがブロックされます（ブランチ保護の設定による）。

| 選択肢 | 意味 | 使いどころ |
|--------|------|-----------|
| **Comment** | フィードバックのみ | 軽微な指摘や質問のみの場合 |
| **Approve** | 承認 | マージしてよいと判断した場合 |
| **Request changes** | 修正要求 | マージ前に修正が必要な場合 |

## 17.4 ブランチ保護ルール

コードレビューの文化をチームに根付かせるためには、仕組みとしてレビューを必須にすることが効果的です。GitHubの**ブランチ保護ルール（Branch Protection Rules）**を使うと、mainブランチへの直接pushを禁止し、Pull Requestを経由したレビュー付きのマージを強制できます。

### 設定方法

ブランチ保護ルールは、リポジトリの設定画面から構成します。

1. リポジトリのページで**Settings**タブを開きます
2. 左のメニューから**Branches**を選択します
3. 「**Add branch protection rule**」（または「Add branch ruleset」）をクリックします
4. 「Branch name pattern」フィールドに保護したいブランチ名を入力します（例：`main`）

<!-- screenshot: ブランチ保護ルールの設定画面 -->

### 主な設定項目の解説

ブランチ保護ルールには多くの設定項目がありますが、ここでは特に重要なものを一つずつ解説します。

**Require a pull request before merging（マージ前にPull Requestを必須にする）**は、最も基本的な保護ルールです。この設定を有効にすると、保護されたブランチへの直接pushができなくなり、必ずPull Requestを経由する必要があります。これにより、すべての変更がレビュー可能な状態になります。

**Require approvals（承認を必須にする）**は、Pull Requestのマージに必要な承認レビューの数を指定します。1人のレビュアーの承認を必須とするのが最も一般的ですが、セキュリティが重要なプロジェクトでは2人以上を求める場合もあります。承認数は多いほど安全ですが、開発スピードとのバランスを考える必要があります。小〜中規模のチームでは1〜2人が適切でしょう。

**Dismiss stale pull request approvals when new commits are pushed（新しいpush後に承認をリセットする）**は、レビュー承認後にPR作成者が新しいコミットをプッシュした場合、以前の承認を無効にする設定です。この設定がないと、承認を得た後にレビューされていないコードを追加してマージできてしまいます。セキュリティの観点から、この設定は有効にすることを強く推奨します。

**Require review from Code Owners（Code Ownersのレビューを必須にする）**は、前章で学んだCODEOWNERSファイルと連携する設定です。変更されたファイルのCode Ownerとして指定されたメンバーの承認がないとマージできなくなります。

**Require status checks to pass before merging（マージ前にステータスチェックの成功を必須にする）**は、CI/CDパイプライン（GitHub Actionsなど）の結果をマージ条件にする設定です。テストが失敗した状態のコードがマージされることを防ぎます。第12章で学んだGitHub Actionsと組み合わせると特に効果的です。

**Require branches to be up to date before merging（マージ前にブランチの最新化を必須にする）**は、Pull Requestのブランチがmainブランチの最新の状態と統合されていることを要求する設定です。古いmainブランチに基づいたコードがマージされて問題が発生するのを防ぎます。

**Include administrators（管理者にもルールを適用する）**は、リポジトリの管理者であっても保護ルールの例外にしない設定です。「管理者だから直接pushしてもいい」という例外を許すと、ルールの意味が薄れてしまいます。チーム全体の規律を保つために、この設定を有効にすることを推奨します。

| 設定項目 | 説明 | 推奨 |
|---------|------|------|
| **Require a pull request** | PR経由のマージを必須 | 有効 |
| **Require approvals** | 承認レビューの数 | 1〜2人 |
| **Dismiss stale approvals** | 新push後に承認をリセット | 有効 |
| **Require Code Owner reviews** | Code Ownerのレビュー必須 | チーム規模による |
| **Require status checks** | CI成功を必須 | 有効 |
| **Require up to date branch** | 最新mainとの統合必須 | 有効 |
| **Include administrators** | 管理者にも適用 | 有効 |

### CLIでの設定

ブランチ保護ルールは、GitHub APIを使ってCLIから設定することも可能です。

```bash
# CLIでブランチ保護を設定する例
gh api repos/{owner}/{repo}/branches/main/protection -X PUT \
  -f "required_pull_request_reviews[required_approving_review_count]=1" \
  -F "required_pull_request_reviews[dismiss_stale_reviews]=true" \
  -F "enforce_admins=true" \
  -F "required_status_checks=null" \
  -F "restrictions=null"
```

## 17.5 レビュアーとしてのベストプラクティス

コードレビューの質は、レビュアーの姿勢と技術に大きく左右されます。ここでは、効果的なレビュアーになるためのベストプラクティスを紹介します。

### 建設的なフィードバックを心がける

レビューのコメントは、批判ではなく改善提案として書きましょう。「このコードはダメだ」ではなく「こうすると可読性が上がるかもしれません」のように、具体的な改善の方向性を示すことが大切です。

人はコードに自分のアイデンティティを投影しがちです。「あなたのコードが悪い」は「あなたが悪い」と受け取られることがあります。主語を「コード」にして、「この部分」「このロジック」のように対象を限定すると、フィードバックが受け入れられやすくなります。

### コメントのプレフィックスを活用する

レビューコメントの重要度が分からないと、PR作成者はすべてのコメントに同じ重みで対応しようとして疲弊してしまいます。以下のようなプレフィックスを使って、コメントの性質を明示しましょう。

```markdown
[Must] ここにSQLインジェクションの脆弱性があります。
ユーザー入力をそのままクエリに含めるのは危険です。
プレースホルダーを使ったパラメータ化クエリに変更してください。

[Nit] 変数名は camelCase に統一した方がコード全体の一貫性が保てそうです。
ここだけ snake_case になっています。

[Question] この実装でMapではなくListを選んだ理由を教えてください。
キーによる検索が多い場面ではMapの方が効率的かもしれないと感じました。

[Praise] このエラーハンドリングのパターン、とても分かりやすいです！
今後のコードでも参考にしたいと思います。
```

**Must（必須）**は、マージ前に必ず修正が必要な問題です。セキュリティの脆弱性、データ損失のリスク、明確なバグなどがこれに当たります。

**Nit（些細な指摘）**は、修正が望ましいが、必須ではない軽微な問題です。命名規則のずれ、コードスタイルの微調整、より良い書き方の提案などです。PR作成者はNitを後回しにしたり、別のPRで対応したりできます。

**Question（質問）**は、コードの意図や設計判断について確認するための質問です。レビュアー自身の理解を深めるためのものであり、必ずしも修正を求めているわけではありません。

**Praise（称賛）**は、良いコードを見つけたときに積極的に書きましょう。問題点だけを指摘するレビューは、受ける側のモチベーションを下げてしまいます。良い点を認めることで、チーム内にポジティブなレビュー文化が育ちます。

### 迅速にレビューする

Pull Requestが長時間レビューされずに放置されると、開発の流れが停滞します。理想的には、レビューリクエストから24時間以内に最初のフィードバックを返すことを目標にしましょう。すべてを詳細に見る時間がない場合でも、「今日中に確認します」という一言を返すだけで、PR作成者の不安が和らぎます。

### 全体像を把握してからレビューする

行単位でコードを読む前に、まずPull Requestの説明文を読み、変更の全体像を把握しましょう。何を目的とした変更なのか、どのような設計判断があるのかを理解した上でコードを読むと、的確なフィードバックが書けます。

## 17.6 PR作成者としてのベストプラクティス

レビューの質は、レビュアーだけでなくPR作成者の工夫にも大きく依存します。レビューしやすいPull Requestを作ることは、PR作成者の責任であり、チーム全体の生産性を高める重要なスキルです。

### 小さなPull Requestを作る

Pull Requestのサイズは、レビューの質に直結します。100行の変更であれば、レビュアーは一つひとつの行を注意深く確認できます。しかし、1000行を超えるPull Requestでは、集中力が続かず、重要な問題を見逃す確率が高くなります。

目安として、**変更は400行以内に収める**ことを心がけましょう。もし大きな機能を開発する場合は、複数のPull Requestに分割することを検討してください。たとえば、データベースのスキーマ変更、バックエンドのAPI実装、フロントエンドのUI実装をそれぞれ別のPull Requestとして送ることで、各PRのレビューが容易になります。

### セルフレビューを行う

Pull Requestを提出する前に、自分自身でFiles changedタブを確認しましょう。意外なほど多くの問題が、この段階で見つかります。デバッグ用のprint文の消し忘れ、コメントアウトしたままのコード、不要なファイルの追加など、セルフレビューで発見できる問題は少なくありません。

セルフレビューのもうひとつの効果は、レビュアーの負担を減らすことです。明らかな問題がない状態でレビューに出すことで、レビュアーはより本質的な設計やロジックの議論に集中できます。

### テストを含める

Pull Requestにテストコードを含めることは、レビュアーに大きな安心感を与えます。テストがあれば、レビュアーは「この変更がどのような動作を期待しているのか」をテストコードから読み取ることができます。また、CIでテストが通っていることを確認できれば、基本的な動作に問題がないことが保証されます。

新しい機能を追加するPRには対応するテストを、バグ修正のPRにはそのバグを再現するテストを含めましょう。テストなしのPull Requestは、レビュアーに「本当にこの変更は正しく動作するのだろうか」という不安を抱かせます。

### 説明文を丁寧に書く

Pull Requestの説明文（description）は、レビュアーへの手紙のようなものです。何を変更したのか、なぜその変更が必要なのか、どのようにテストしたのかを明確に書きましょう。関連するIssue番号があれば `Closes #42` のように記載して紐付けます。スクリーンショットやGIFアニメーションで変更の効果を視覚的に示すのも効果的です。

### コンテキストを共有する

変更の背景にある設計上の判断や、検討した代替案があれば、PRの説明文やコメントで共有しましょう。「AではなくBを選んだ理由は〜」という情報があると、レビュアーは同じ議論を繰り返さずに済みます。

## 17.7 まとめ

この章では、コードレビューの目的、GitHubでの具体的な操作方法、そしてレビュアーとPR作成者それぞれのベストプラクティスについて学びました。重要なポイントを振り返りましょう。

- コードレビューは、**バグの早期発見、知識の共有、品質の向上、一貫性の維持**のために行う
- GitHubでは**Files changedタブ**で差分を確認し、行コメントやsuggestion機能を使ってフィードバックを提供する
- レビューの結論として**Comment（コメント）、Approve（承認）、Request changes（変更要求）**の3種類を選択できる
- **ブランチ保護ルール**を設定することで、レビューを開発プロセスに組み込み、mainブランチの品質を保つことができる
- レビュアーは**建設的なフィードバック**を心がけ、コメントには**Must/Nit/Question/Praise**などのプレフィックスで重要度を明示する
- PR作成者は**小さなPR、セルフレビュー、テストの付与、丁寧な説明文**を心がけることで、レビューの質と速度を向上させる

コードレビューは技術的なプラクティスであると同時に、チームのコミュニケーションの場でもあります。互いを尊重し、学び合う姿勢で取り組むことで、チーム全体の成長につながります。
